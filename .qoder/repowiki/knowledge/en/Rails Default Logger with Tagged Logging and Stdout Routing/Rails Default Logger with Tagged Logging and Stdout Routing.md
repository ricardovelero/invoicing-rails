The application uses Rails' built-in logging stack with no third-party logger gem. All log output is produced through ActiveSupport::Logger (the default Rails logger) and routed via ActiveSupport::TaggedLogging. There are no custom logger classes, structured-log libraries, or dedicated logging modules in the codebase.

Framework and configuration
- config/application.rb loads defaults for Rails 7.1 and does not override the logger, so the standard Rails logger is used everywhere.
- config/environments/development.rb leaves the default file-based logger active, enables verbose query logs (config.active_record.verbose_query_logs = true), suppresses asset-request noise (config.assets.quiet = true), and routes deprecations to the Rails logger (config.active_support.deprecation = :log).
- config/environments/test.rb sends deprecations to stderr (config.active_support.deprecation = :stderr) and otherwise relies on the default test logger.
- config/environments/production.rb sets config.log_level = :debug, adds a request_id tag via config.log_tags = [:request_id], pins the formatter to ::Logger::Formatter.new, and conditionally replaces the logger when RAILS_LOG_TO_STDOUT is set by wiring an ActiveSupport::Logger backed by $stdout into ActiveSupport::TaggedLogging. A commented-out block shows the intended path for syslog integration in distributed setups.

Log structure and fields
- Log lines are plain-text formatted by Ruby's Logger::Formatter; there is no JSON or other structured format configured.
- The only contextual field injected automatically is request_id (via config.log_tags = [:request_id] in production). No custom tags or structured fields are added anywhere in the app.

Usage in application code
- A grep across all .rb files finds zero calls to Rails.logger, logger.debug, logger.info, etc. Application controllers, models, jobs, mailers, and helpers do not emit any explicit log statements — they rely entirely on Rails framework-generated request, SQL, cache, and deprecation logs.

Sinks
- Development: file-based logs under log/development.log (and log/test.log, log/production.log as generated by Rails).
- Production: stdout when RAILS_LOG_TO_STDOUT is present (suitable for containerized environments like Render); otherwise falls back to the default file sink. Syslog is documented but not enabled.

Rules developers should follow
- Do not add ad-hoc puts / print statements; use Rails.logger if you need custom entries.
- Prefer the existing request_id tag for correlating logs across services; avoid adding unstructured free-form text without context.
- Keep log levels appropriate: info for normal operations, warn for recoverable issues, error for failures that should be surfaced.
- In production, ensure RAILS_LOG_TO_STDOUT is set so logs reach the platform's collector rather than being written to disk.
